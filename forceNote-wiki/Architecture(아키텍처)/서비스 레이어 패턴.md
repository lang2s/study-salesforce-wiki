# 서비스 레이어 패턴

> 비즈니스 로직을 Trigger에서 분리해 재사용 가능한 `ServiceLayer` 클래스에 집중. Trigger → TriggerHandler → ServiceLayer 계층 구조.

---

## 왜 서비스 레이어가 필요한가

비즈니스 로직을 Trigger에 직접 작성하면 세 가지 문제가 생긴다.

1. **재사용 불가** — Trigger에서만 호출되므로, LWC(@AuraEnabled)나 REST API, Flow에서 동일 로직을 사용하려면 코드를 복사해야 한다. 복사된 코드는 버그 수정 시 여러 곳을 동시에 고쳐야 하는 부채가 된다.
2. **테스트 어려움** — Trigger 단위 테스트는 DML을 직접 실행해야 하므로 느리고 격리가 어렵다. ServiceLayer의 static 메서드는 DML 없이 단독 테스트가 가능하다.
3. **가독성 저하** — Trigger가 수백 줄이 되면 이벤트 분기와 비즈니스 로직이 뒤섞여 유지보수가 불가능해진다.

## 언제 도입하나

| 상황 | 권장 |
|---|---|
| 동일 로직이 Trigger·LWC·Flow 등 2개 이상에서 필요 | 서비스 레이어 도입 |
| 비즈니스 로직이 50줄 이상으로 복잡 | 서비스 레이어 도입 |
| 단순 필드 갱신 1개 | Trigger 직접 작성도 허용 |
| ISV 패키지 개발 | 반드시 도입 (고객 확장성 필요) |

## 주의사항

- **ServiceLayer 메서드는 `static`으로 유지한다.** 인스턴스화 없이 어디서든 호출할 수 있어야 재사용성이 보장된다.
- **Handler에서 직접 DML을 실행하지 않는다.** DML은 ServiceLayer 또는 `Safely` 클래스에서만 수행해 공유 규칙 적용 일관성을 유지한다.
- **Before Trigger에서 ServiceLayer를 호출하면 DML이 없어야 한다.** Before Trigger에서는 `Trigger.new`를 in-memory로 수정하면 되고, DML을 실행하면 추가 Trigger 체인이 발생한다.
- **순환 호출 주의** — ServiceLayer가 DML → Trigger → ServiceLayer를 재귀 호출하는 루프가 생기지 않도록 설계한다.

---

## 계층 구조

```
Trigger (1줄)
    ↓
TriggerHandler (이벤트 분기)
    beforeInsert / afterInsert / ...
    ↓
ServiceLayer (비즈니스 로직)
    static 메서드 — Trigger/LWC/Flow/REST 모두 호출 가능
    ↓
DML (Safely.cls 또는 insert as user)
```

## AccountServiceLayer 예시

```apex
public with sharing class AccountServiceLayer {

    // Trigger에서 호출
    public static void manipulateAccountsWithTriggerLogic(
            List<Account> accounts,
            Map<Id, Account> oldAccounts) {

        for (Account acct : accounts) {
            Account oldAcct = oldAccounts?.get(acct.Id);
            if (acct.ShippingCity != oldAcct?.ShippingCity) {
                acct.BillingCity = acct.ShippingCity; // in-memory 수정
            }
        }
        // Before trigger: DML 없음 — Trigger.new가 직접 반영됨
    }

    // LWC/Flow에서도 동일 메서드 호출 가능
    public static List<Account> safelyCreateAccounts(List<Account> accounts) {
        List<Database.SaveResult> results =
            new Safely()
                .allOrNothing()
                .throwIfRemovedFields()
                .doInsert(accounts);
        // 결과 처리...
        return accounts;
    }
}
```

## TriggerHandler에서 ServiceLayer 호출

```apex
public class AccountTriggerHandler extends TriggerHandler {

    private List<Account> newAccounts;
    private Map<Id, Account> oldAccountsMap;

    public AccountTriggerHandler() {
        this.newAccounts    = (List<Account>) Trigger.new;
        this.oldAccountsMap = (Map<Id, Account>) Trigger.oldMap;
    }

    public override void beforeUpdate() {
        AccountServiceLayer.manipulateAccountsWithTriggerLogic(
            newAccounts,
            oldAccountsMap
        );
    }

    public override void afterInsert() {
        AccountServiceLayer.someAfterInsertLogic(newAccounts);
    }
}
```

## 설계 원칙

| 원칙 | 설명 |
|---|---|
| Trigger는 1줄 | `new AccountTriggerHandler().run()` |
| Handler는 분기만 | 비즈니스 로직 없이 ServiceLayer 위임 |
| ServiceLayer는 static | 어디서든 호출 가능 (Trigger, @AuraEnabled, REST, Flow) |
| with sharing | 공유 규칙 적용 (inherited sharing도 허용) |
| DML은 ServiceLayer에서 | Handler에서 직접 DML 금지 |

## @AuraEnabled에서 ServiceLayer 재사용

```apex
@AuraEnabled(cacheable=true)
public static List<Account> getAccountsForUI() {
    try {
        // ServiceLayer의 쿼리 로직 재사용
        return AccountServiceLayer.getActiveAccounts();
    } catch (Exception e) {
        throw new AuraHandledException(e.getMessage());
    }
}
```
