## 1. 핸드오프 계약 (스쿼드 → 스쿼드)

받는 쪽은 아래 "수락 조건"을 만족하지 않는 핸드오프를 **반려**한다(다음 단계로 진행 금지). 반려 시 책임은 **넘기는 쪽**이 진다.

| From → To | 넘기는 산출물 | 받는 쪽 수락 조건 | 불충분 시 |
|---|---|---|---|
| 기획 → 소스 | 리서치 플랜(주제·예상 소스·범위 제약) | 소스 위치 추정이 있고, 범위 제약("~빼고")이 명시됨 | 소스로 반려 → 기획 재작성 |
| 소스 → 작성 | 소스맵 + 전수 추출 덤프 + coverage-checker 보고 | 추출이 **요약 아님**(전수), 추가 소스 후보가 평가됨 | 작성으로 반려 → 소스 재추출 |
| 작성 → 검증 | 생성/수정 파일 + 추출 항목 목록 + 의도적 제외 목록 | 파일이 구조 체크리스트 통과, 작성시점 역링크 반영됨 | 검증으로 반려 → 작성 수정 |
| 검증 → 탐색·정비 | pass/fail + 갭/오류 리포트 | completeness ✅ + source-verifier ✅ (둘 다 통과) | 탐색으로 넘기지 않음 → 작성 재작업 |
| 탐색·정비 → 조율 | 샤드·MOC·index 갱신 + 양방향 링크 완료 보고 | 새 파일이 샤드 1곳 등재, 형제 역링크 점검됨 | 조율로 넘기지 않음 → 정비 보완 |

> **계약의 핵심:** seam마다 *받는 쪽이 게이트키퍼*다. 불완전하면 받지 말고 돌려보낸다 → 넘기는 쪽이 끝까지 책임진다.

## 2. 책임 지도 (RACI — 반복 책임별 주담당 / 백업)

지도에 **없는** 일 = 무주공산 → 3번 절차로 즉시 행 추가. 주담당이 1차, 백업이 그물.

| 반복 책임 | 주담당 (1차) | 백업 (그물) | 트리거 |
|---|---|---|---|
| 양방향 링크 보장 | 작성 (writer, 작성 시점) | 탐색·정비 (cross-linker, 사후) | 새 파일·보완 |
| 깨진 wikilink 탐지 | 탐색·정비 (wiki-linter) | — | /lint |
| 깨진 링크 수정 | 작성 (콘텐츠) / 탐색·정비 (탐색 파일) | 조율(PM 배정) | lint 발견 후 |
| 키워드 샤드 등재 | 탐색·정비 (index-manager) | — | 새 파일 |
| 샤드 상한 분할 | 탐색·정비 (index-manager) | 조율 (회고가 감지) | ~300줄 초과 |
| 소스 누락 탐지 | 소스 (coverage-checker) | 조율 (회고 모드 B) | 작성과 병렬 |
| GA 기능 후속 네임스페이스 작성 | 기획 (백로그 등재) | 조율 (회고가 추적) | 릴리즈 노트 후 |
| 백로그 ✅ 항목 아카이브 이동 | 조율 (회고) | 조율 (PM) | ✅ 처리 시 |
| 백로그 비대화 점검 | 조율 (회고) | — | 모드 B |
| Tier 3 경고 블록 | 작성 | 검증 (source-verifier) | 외부지식 소스 |
| 챕터 개요→서브페이지 링크 | 작성 | 탐색·정비 (cross-linker) | 서브페이지 분화 |
| 파일 위치 적절성 | 탐색·정비 (wiki-linter #8) | 조율 (회고) | /lint |

## 3. 무주공산 처리 (지도에 없는 일이 나왔을 때)

```
1. 회고(또는 누구든) seam에서 샌 일을 발견
2. 이 SEAM_MAP 2번 표에 행을 추가하며 주담당+백업 배정
3. 배정 전까지는 PM이 임시 소유 (방치 금지)
4. 동시에 그 일이 샌 근본 원인을 해당 스쿼드 헌장 또는 에이전트 .md에 방지 규칙으로 추가
```

> 핵심: **"내 일 아님"으로 끝나지 않는다.** 주인이 없으면 만든다. 만들 때까지 PM이 든다.

## 4. 회고 seam 순찰 체크리스트 (매 회고 사이클)

wiki-retrospective가 모드 A/B에서 아래를 점검한다 — 팀을 가로질러 도는 유일한 능동 장치.

```
□ 이번 작업에서 핸드오프 반려가 있었나? → 어느 계약(1번 표)이 약했나 → 계약 문구 강화
□ 어느 스쿼드도 '내 일'이라 안 집은 항목이 있었나? → 2번 표에 행 추가 + 주담당 배정
□ 같은 종류의 누락이 2회 이상 재발했나? → 주담당 스쿼드 헌장/에이전트 .md에 방지 규칙 추가
□ 백업(그물)이 자꾸 1차를 대신 메우고 있나? → 1차 주담당의 규칙이 약하다는 신호 → 1차 강화
```

## 알려진 seam 사고 (씨앗 사례 — 이 지도가 막으려는 것)

| 사고 | 샜던 이유 | 지금 막는 행 |
|---|---|---|
| 단방향 링크 누적 | writer는 만들고 cross-linker만 사후 수거 → 작성 시점 무책임 | 2번 "양방향 링크 보장" 주담당=작성 |
| 백로그 비대화 | ✅ 항목을 아무도 아카이브 안 함 | 2번 "아카이브 이동" 주담당=회고 |

## 규칙 쌍 일관성 잠금 (rule-pair locks)

서로 다른 스쿼드가 소유한 규칙이 **같은 예외를 공유**하면, 한쪽만 바꿀 때 모순이 생긴다 — *한 스쿼드가 의도해서 만든 걸 다른 스쿼드가 결함으로 잡는다.* 아래 쌍은 **항상 함께** 바꾼다.

| 규칙 A (소유 스쿼드) | 규칙 B (소유 스쿼드) | 공유 예외 | 어기면 |
|---|---|---|---|
| writer 작성 시점 역링크 (작성) | wiki-linter 단방향 검사 #6 (탐색·정비) | spoke→hub 단방향은 정상 | linter가 writer 의도 단방향을 결함으로 재검출 → 가짜 경고 더미에 진짜 결함이 묻힘 |

> 4번 seam 순찰에 포함: 규칙을 추가·수정할 때 **"이 규칙과 짝을 이루는 검사가 다른 스쿼드에 있나?"** 를 확인하고, 있으면 같이 고친다.
