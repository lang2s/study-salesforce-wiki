# 비동기 컨텍스트 선택 매트릭스

> 언제 `@future` vs `Queueable` vs `Batch` vs `Scheduled` vs `Database.Cursor`를 쓸지 결정하는 가이드.

---

## 결정 매트릭스

| 상황 | 권장 | 이유 |
|---|---|---|
| 단순 비동기, 객체 불필요 | `@future` | 가장 단순 |
| HTTP Callout이 필요한 비동기 | `@future(callout=true)` | @future의 유일한 callout 방법 |
| 객체/컬렉션을 파라미터로 전달 | `Queueable` | @future는 primitives만 허용 |
| Callout + 복잡한 후처리 | `Queueable + Database.AllowsCallouts` | 체이닝 가능, 파라미터 자유 |
| 작업 순서 제어(체이닝) 필요 | `Queueable` 체이닝 | execute당 1개만 enqueue |
| 대용량 데이터(수만 건+) 처리 | `Batch Apex` | QueryLocator로 힙 한도 우회 |
| execute() 간 상태 유지 필요 | `Batch + Database.Stateful` | 없으면 각 execute마다 변수 초기화 |
| 정기 실행 (cron) | `Schedulable` | 배치 스케줄은 System.scheduleBatch |
| 페이지 기반 대용량 레코드 이동 | `Database.Cursor + Queueable` | 최대 5천만 건, position 제어 가능 |

---

## @future 제약

```apex
// ✅ 기본 @future — Primitives만 파라미터로 받을 수 있음
@future
public static void processData(String data, Integer count) { }

// ✅ Callout 허용
@future(callout=true)
public static void callExternalApi(String endpoint) { }

// ❌ 불가 — SObject, List<SObject> 파라미터 없음
// @future public static void process(List<Account> accts) { } // 컴파일 에러
```

> [!warning] @future 한계
> - SObject/List를 파라미터로 받을 수 없음 → Id를 받아 내부에서 쿼리
> - 체이닝 불가 (@future 안에서 @future 호출 금지)
> - 호출 횟수 한도: 트랜잭션당 50개

## Database.Cursor — 페이지 기반 대용량 처리 (Summer '24 GA)

> SOQL 결과를 커서 포인터로 열어두고 `fetch(position, batchSize)`로 페이지 단위 처리. Batch Apex 없이도 최대 5천만 건 처리 가능.

```apex
public with sharing class QueryChunkingQueueable implements Queueable {
    private Database.Cursor locator;
    private Integer position;

    public QueryChunkingQueueable() {
        locator = Database.getCursor(
            'SELECT Id FROM Contact WHERE LastActivityDate = LAST_N_DAYS:400',
            AccessLevel.USER_MODE);
        position = 0;
    }

    public void execute(QueueableContext ctx) {
        Integer remainingRows = locator.getNumRecords() - position;
        if (remainingRows == 0) {
            return; // Nothing to do
        }

        // Take the minimum of batch size and remaining rows to avoid over-fetching
        Integer fetchSize = Math.min(200, remainingRows);

        List<Contact> scope = locator.fetch(position, fetchSize);
        position += scope.size();
        // do something, like archive or delete the scope list records
        if (position < locator.getNumRecords()) {
            // process the next chunk
            System.enqueueJob(this);
        }
    }
}
```

| API | 시그니처 | 설명 |
|---|---|---|
| `Database.getCursor(query, accessLevel)` | `Database.Cursor` | SOQL 문자열로 커서 생성 |
| `Database.getCursorWithBinds(query, bindMap, accessLevel)` | `Database.Cursor` | 바인드 변수 포함 커서 생성 |
| `locator.fetch(position, batchSize)` | `List<SObject>` | position 위치부터 batchSize만큼 조회 |
| `locator.getNumRecords()` | `Integer` | 커서 전체 레코드 수 |

> [!tip] Cursor vs Batch 선택 기준
> - **Database.Cursor**: Queueable 안에서 사용. position 직접 제어 가능. 단순 반복 처리에 적합.
> - **Batch Apex**: 복잡한 집계(`Database.Stateful`), 스케줄링, 실패 시 재시작 지원. 기존 Batch 코드 자산 재사용 시 유리.

> [!warning] 커서 한도
> 커서는 동기·비동기 모두 최대 5천만 행. 커서 내부 `fetch()` 각 호출은 별도 SOQL 쿼리 한도를 소모하지 않는다.

## 선택 흐름도

```
비동기 처리가 필요한가?
  │
  ├─ 단순 + primitives만 필요 → @future
  │     └─ HTTP 호출도 필요? → @future(callout=true)
  │
  ├─ 객체/컬렉션 전달 필요 → Queueable
  │     └─ HTTP 호출도 필요? → Queueable + Database.AllowsCallouts
  │
  ├─ 수만 건 이상 대용량 → Batch Apex 또는 Database.Cursor + Queueable
  │     ├─ 실행 간 집계 필요? → Batch + Database.Stateful
  │     └─ 페이지 position 제어 필요? → Database.Cursor + Queueable
  │
  └─ 정기 실행 필요 → Schedulable
        └─ 배치 정기 실행? → System.scheduleBatch()
```

## 주의사항

> [!warning] 비동기 공통 한도 — 트랜잭션당 제한
> - **@future**: 트랜잭션당 최대 50회 호출. @future 내에서 @future 재호출 금지.
> - **Queueable**: `System.enqueueJob()` 트랜잭션당 최대 50회. execute() 내 체이닝은 1회만 허용.
> - **Batch**: 동시 실행 배치 최대 5개 (Flex Queue 포함 100개 대기). `Database.executeBatch()` 한 트랜잭션에서 최대 5회.
> - **Schedulable**: 예약된 작업 최대 100개 (동시). `System.schedule()` 동일 시간대 중복 주의.
> - **Database.Cursor**: 동기·비동기 모두 최대 5천만 행. 커서는 Queueable 내에서만 사용 권장.
> - **공통**: 비동기 컨텍스트에서도 SOQL 100회, DML 150회 등 Governor Limits 동일 적용.
