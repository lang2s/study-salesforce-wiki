# 멀티 패키지 구조

> 도메인별로 독립 패키지를 분리해 독립 배포·버저닝. automation-components는 9개 패키지.

---

## 개념 — 멀티 패키지가 해결하는 문제

단일 `force-app/` 디렉토리에 모든 소스를 넣으면 팀이 커질수록 세 가지 문제가 생긴다.

1. **배포 단위 분리 불가** — Collections 기능만 업데이트해도 전체 패키지를 재배포해야 한다.
2. **버저닝 불가** — 기능별로 독립된 버전을 관리할 수 없어 "v2 Collections + v1 Data" 조합 설치가 불가능하다.
3. **고객별 선택 설치 불가** — AppExchange 고객에게 필요한 모듈만 골라서 제공할 수 없다.

멀티 패키지 구조는 도메인(Collections, Data, Flows 등)별로 독립 Unlocked Package를 만들어 이 세 문제를 해결한다.

## 언제 멀티 패키지를 선택하나

결정 매트릭스는 [아래 표](#싱글-vs-멀티-패키지-결정-매트릭스)를 참조. 핵심 기준은 **"도메인이 독립 배포·독립 버저닝이 필요한가"**이다. 도메인 간 의존성이 깊으면 멀티 패키지로 전환해도 버전 충돌이 잦아 오히려 복잡도가 높아진다.

## 패키지 의존성과 주의사항

- **패키지 간 의존성 방향은 단방향을 유지한다.** `src-flows`가 `src-collections`를 참조하는 것은 허용되지만, 순환 참조(`A → B → A`)는 설치 실패를 유발한다.
- **`ancestorVersion: HIGHEST`** 설정은 항상 최신 배포 버전을 조상으로 삼는다. 특정 버전으로 고정하려면 `ancestorVersion: "60.0.0-1"` 형식으로 명시한다.
- **설치 키(installation-key)** 는 패키지를 공개 배포할 때 필수. 내부용이라도 관리 목적으로 설정을 권장한다.
- **패키지 ID(`0Ho...`)는 변경 불가.** Alias를 잘못 매핑하면 잘못된 패키지에 버전이 생성된다. `packageAliases` 섹션은 처음 생성 시 자동 삽입되며 수동 수정은 금지한다.

## 폴더 구조

```
automation-components-main/
├── src-collections/
│   ├── main/default/classes/   ← 소스
│   └── test/default/classes/   ← 테스트 분리
├── src-data/
│   ├── main/default/classes/
│   └── test/default/classes/
├── src-flows/
│   ├── main/default/classes/
│   ├── main/default/lwc/
│   └── test/default/classes/
└── sfdx-project.json
```

> `main/`과 `test/`를 분리해 테스트 클래스만 별도 관리.

## 싱글 vs 멀티 패키지 결정 매트릭스

| 상황 | 선택 |
|---|---|
| 단일 앱, 팀 하나 | 싱글 패키지 (`force-app/`) |
| 도메인 독립 배포 필요 | 멀티 패키지 |
| 외부 고객에게 일부만 배포 | 멀티 패키지 |
| 재사용 라이브러리 배포 | 멀티 패키지 |
| AppExchange 관리 패키지 | 멀티 패키지 필수 |

## 패키지 버전 관리

```bash
# 패키지 버전 생성
sf package version create --package "AC - Collections" --code-coverage --installation-key test1234

# 버전 설치
sf package install --package "AC - Collections@60.0.0-1" --target-org my-org

# NEXT → 자동 증가 (60.0.0.NEXT → 60.0.1 → 60.0.2 ...)
```

| 버전 표기 | 의미 |
|---|---|
| `60.0.0.NEXT` | 다음 빌드 번호 자동 증가 |
| `HIGHEST` | 배포된 가장 높은 버전을 조상으로 |
